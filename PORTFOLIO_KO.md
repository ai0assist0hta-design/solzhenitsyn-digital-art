## 1. 컨셉

이 프로젝트는 알렉산드르 솔제니친의 핵심 주제들 — 거짓의 구조, 정당화로서의 이데올로기, 수용소 체제, 양심, 저항, 그리고 개인의 비참여 — 에 대한 디지털 독해다. 여섯 개의 주제는 각각 **읽는 것이 아니라 만지는** 인터랙티브 오브젝트로 구현되었다.

작품 전체를 관통하는 시각 장치는 **redacted 문서**다. 본문 대부분이 검은 블록으로 가려져 있고, 하중을 지탱하는 키워드 몇 개만이 빨간색으로 노출된다. 이것은 단순한 스타일 장식이 아니라 **주제 그 자체**다. 거짓 위에 서 있는 체제는 구조적으로 무엇을 검열해야 하는지 아는 체제다. 관람자는 여섯 개의 그런 문서를 통과하며, 각 문서는 서로 다른 물리 기반 오브젝트로 갈라진다.

미학적 선택은 **브루탈리즘**이다. 흰색/검은색 배경, 모노스페이스와 인터 타이포그래피, 단일 강조색(`#dc143c`), 8-column 그리드, 자기 자리를 정당화하지 못하는 어떤 장식도 없는 구성.

## 2. 디자인 번역: 텍스트에서 오브젝트로

나는 우선 그래픽 디자이너이고, 이 프로젝트는 코드로 수행된 그래픽 디자인 작업이다. 기술 스택은 디자인 의도를 위해 학습된 것이지 그 반대가 아니다. 모든 시각적 결정은 단 하나의 질문에서 시작했다 — **이 특정 주제의 시각적 논리는 무엇인가?** 솔제니친의 텍스트가 내용을 줬다. 내 일은 그것을 운반할 수 있는 시각 시스템을 발명하는 것이었다.

### 2.1. 오브젝트 이전에 시스템을 잠근다
어떤 오브젝트도 그리기 전에 시스템부터 잠갔다. 그 외 모든 것을 정의한 6개의 결정:

- **단일 강조색: `#dc143c` (crimson).** 기업적 빨강도, 경고 빨강도 아닌 — 검열관의 펜 색, 마른 피의 색, 종이를 뚫을 만큼 강하게 그어진 밑줄의 색. redaction을 통과해 살아남은 모든 키워드가 이 색이다. 모든 인터랙티브 요소가 이 색에 반응한다. 두 번째 강조색은 없다.
- **8-column 그리드.** 편집 디자인이 아니라 건축적이다. 본체는 가운데 살고, 사이드바(info-active 시)가 양쪽에서 각각 2 컬럼을 차지하여 중앙에 4 컬럼의 호흡 공간을 남긴다. 그리드는 잡지 펼침면이 아니라 건물이다.
- **두 개의 서체, 두 개의 목소리.** JetBrains Mono가 문서 톤을 담당한다 — 타자기처럼, 법정 기록처럼, *철해진 어떤 것*처럼 보인다. Inter가 UI 톤을 담당한다 — 어디를 누르라고 알려주는 동시대 목소리. 이 둘을 섞은 것은 의도적이다. 당신은 새 인터페이스를 통해 오래된 문서를 읽고 있다.
- **검은 █ 블록을 주된 redaction 시각 장치로.** 기밀 해제된 FOIA 문서에서 가져온 것이지 발명한 것이 아니다. █ 글자는 주제적 작업을 수행한다 — 체제가 우리에게 보여주고 싶지 않았던 것의 문자 그대로의 자국이다.
- **장식 없음.** 브루탈리즘의 첫 번째 계명은 기능적 정직이다. 자기 자리를 정당화하지 못하는 어떤 것도 작품이 무엇인가에 대한 거짓이다.
- **두 가지 상태: 라이트와 다크.** 라이트 = "출판된 문서". 다크 = "실제로 살아있는 문서" — 밤에, 사적 독해 속에서, 주변 세계가 벗겨진 상태로. 스타일적 토글이 아니다. 독해 모드 토글이다.

### 2.2. 각 주제를 시각 오브젝트로 번역
시스템이 잠긴 후, 6개의 주제 각각에 대한 질문은 동일했다 — **이 아이디어를 운반하는 가장 단순한 시각 구조는 무엇인가?** 각 주제에 대해 내가 나 자신에게 쓴 브리프:

| 주제 | 문학적 명제 | 내가 선택한 시각 논리 |
|---|---|---|
| 1. 거짓의 구조 | "거짓은 콘텐츠가 아니라 인프라다." | 전봇대. 전봇대는 신호를 운반하지, 생성하지 않는다. 빨간 키워드가 구조적 골격이 된다. 읽히지 않는 문장들이 거기에 매달린다. |
| 2. 정당화로서의 이데올로기 | "이데올로기는 정당화이다 — 이성처럼 들릴 때까지 반복된." | CRT 스크린의 벽. 각 스크린이 다른 공식 텍스트를 스크롤. 어떤 논증도 끝나지 않으므로 각 스크린 뒤로 홀로그램 잔상이 뻗어 있다. |
| 3. 수용소 | "철해질 수 있는 숫자로 환원된 한 생애." | 빨간 핵(파일실) 주변을 공전하는 0과 1의 구체. 디지트와 핵 사이의 연결선은 몸과 장부의 링크. |
| 4. 양심 | "συν + εἶδος — 함께-앎. 자아가 자아를 본다." | 서로 다른 속도로 함께 회전하는 동심원 링의 vortex. 모래시계의 허리가 판단의 순간 — 층들이 잠시 만나는 곳. |
| 5. 저항 | "거짓의 힘은 빌려온 목소리에 기댄다." | 모든 방향으로 행진하는 슬로건의 포화 합창 필드. 클릭이 침묵의 원을 연다. 합창은 다시 채워지지만, 침묵은 일어났다. |
| 6. 그러나 나를 통해서는 아니다 | "눈은 계속된다. 문제는 그것들이 너를 통해 계속되는가다." | 동공이 커서를 추적하는 Matrix 스타일의 괄호-눈 비. 클릭이 레이저를 안쪽으로 발사한다 — 그 영역의 눈들이 어두워진다. 벽이 다시 구성된다. |

각 번역은 텍스트의 1:1 일러스트레이션이 아니다. **텍스트가 하는 같은 일을 수행하는 시각 구조**다. 전봇대는 "거짓"의 그림이 아니라 — 거짓이 어떻게 움직이는가의 건축이다. 구체는 "수용소"의 그림이 아니라 — 수용소가 한 사람을 어떻게 처리하는가의 구조다. 이것이 가장 많은 사고를 요구한 움직임이고, 이것이 프로젝트를 정의하는 움직임이다.

### 2.3. 시각적 위계
각 오브젝트 안에서 세 단계의 무게가 모든 무거운 작업을 한다:

1. **Crimson 키워드** — 하중을 지탱하는 용어들. 항상 빨갛다. 보이는 텍스트의 약 10%를 절대 넘지 않는다.
2. **검은 █ 블록** — redacted 본문. 모든 텍스트 표면의 압도적 다수.
3. **회색 문서 본문** — 읽을 수 있지만 후퇴된, redaction을 대비로 가독 가능하게 만드는 주변 레이어.

이 3단계 위계 없이는 redaction은 그저 노이즈다. 이게 있으면, 모든 페이지가 명확한 figure-ground 관계를 갖는다 — 시선이 어디에 머물지 안다.

## 3. 6개의 오브젝트

| # | 주제 | 시각 오브젝트 | 인터랙션 |
|---|------|---------------|---------|
| 1 | 거짓의 구조 | 빨간 키워드가 골격을 이루는 3D 전봇대 | 회전, 줌 |

| 3 | 수용소와 감금 | 0/1 디지트가 공전하는 탄성 핵 구체 | 드래그-당김, 탄성 복귀 |

각 오브젝트는 목적별로 만들어졌다. Theme 1은 커스텀 GLB 전봇대 모델을 사용한 Three.js, Theme 4는 Three.js 셰이더 기반 입자 시스템, Theme 5와 6은 성능을 위한 Canvas2D, Theme 3은 DOM 기반 3D 변환, Theme 2는 CRT 스캔라인과 홀로그램 잔상을 위한 WebGL 셰이더로 구성되었다.

## 4. 기술 스택

- **프레임워크 없음.** Vanilla JS만 사용. 빌드 단계 없이 GitHub Pages에 정적 배포 가능.
- **Three.js** — 3D 씬 (Theme 1, 4).
- **Canvas2D + 스프라이트 캐싱** — 입자 수가 많은 씬을 위함. Theme 6의 괄호 스프라이트는 오프스크린 캔버스에 미리 렌더링한 뒤 매 프레임 `drawImage`로 그려, `fillText × N`의 비용을 회피.
- **CSS containment + DPR 스케일링** — Retina 디스플레이에서 선명하게 렌더링하면서 오버드로우를 피함.
- **WebGL 커스텀 uniforms** (`uIsLightMode`, `uTime`, `uCamZ`) — Theme 2 CRT 스캔라인과 Theme 4 양심 vortex.
- **8-column CSS Grid** — 사이트 전체의 레이아웃 기본 단위.
- **GitHub Pages** 호스팅. 백엔드 없음, DB 없음, 분석툴 없음.

전체 표면적은 두 개 파일에 약 5,000 라인의 JS, 약 1,500 라인의 CSS.

## 5. 프로세스: 인간 + AI 협업

이 프로젝트는 Claude(Anthropic)와의 긴밀한 협업으로 만들어졌다. 발현된 협업 모델은 "AI가 작업하고 인간이 리뷰한다"가 아니라 **시니어 디자이너가 빠른 주니어 구현자와 페어링하는** 모습에 가까웠다. 비전, 취향, 방향성은 전적으로 인간 측에 있었고, AI의 역할은 의도를 코드로 번역하고 버그를 드러내는 것이었다.

### AI가 잘한 것
- **구현 속도.** 비주얼 방향이 결정되면, AI는 수백 줄의 코드를 분 단위로 구현했다. 6개 테마를 며칠 안에 완성하는 일정은 단독으로는 불가능했다.
- **내가 지시한 횡단적 리팩터.** 이름 변경, 죽은 코드 제거, 일관성 패스(예: "6개 테마 모두 redacted 블록 형식 통일해")를 내가 지시하면 AI는 grep/sed 수준의 변경을 빠르게 실행했다.
- **내가 지정한 리서치 경로 따라가기.** AI는 내가 어느 실을 잡아당겨야 할지 알고 있을 때 유용한 도구였다. 내가 어디를 보라고 하면, 거기 있는 것을 가져왔다.

### AI가 못한 것
- **미학적 판단.** AI는 능숙하지만 명백히 AI가 쓴 듯한 텍스트를 반복적으로 생산했다. 3개짜리 리스트("violence, obedience, falsehood"), "X is not Y, it's Z" 반전, 과도한 em-dash, 그리고 모든 문단을 자기 요약 격언으로 끝내는 경향. 이런 것들은 손으로 반복해서 빼내야 했다.
- **종교적 / 감성적 클리셰.** 모델은 문서 헤더에 *CC: GOD*, 수신인으로 *every heart* 같은 표현을 기본값으로 생성했다. 둘 다 더 차갑고 견고한 대안(*MEMORY*, *BYSTANDERS*)으로 교체해야 했다.
- **멈춰야 할 때를 모름.** AI는 모든 내러티브 패널 끝에 빨간 이탤릭의 솔제니친 인용구를 장식처럼 추가하는 경향이 있었다. 세 번째쯤부터 패턴이 두드러져, 6개 모두에서 삭제했다.

### 작업 방식이 바뀐 지점
디자이너/엔지니어 커뮤니티에서 일관되게 관찰되는 것 — Hacker News, `r/webdev`, `r/programming`, Simon Willison 등의 글에서 반복적으로 등장 — 은 **AI가 병목을 타이핑에서 결정으로 옮긴다**는 점이다. 변형을 만드는 비용이 거의 0에 수렴하기 때문에, 인간의 일은 변형들 중에서 **선택하고 대부분을 거절**하는 일이 된다. 이 프로젝트에서도 그대로였다. AI가 생성하고, 인간이 무엇을 남길지 골랐다.

두 번째로, 역시 널리 공유되는 관찰은 **AI는 강한 비전을 가진 사람에게는 곱셈자(force multiplier)가 되지만, 비전이 없는 사람에게는 희석제가 된다**는 점이다. 명확한 컨셉(브루탈리스트 redaction, 솔제니친-as-주제)을 가진 상태에서 AI는 모든 것을 가속했다. 컨셉이 없었다면, AI는 평범한 랜딩 페이지 6개를 만들었을 것이다.

## 6. 구체적 도전 과제 — 내가 어떻게 접근했는가

이 프로젝트의 버그와 디자인 문제들은 "AI가 발견한 것"이 아니다. **내가** 알아챘고, 어디를 봐야 할지에 대한 가설을 세웠고, AI에게 어느 흔적을 따라가라고 정확히 지시했고, 돌아온 결과를 판단했다. AI는 손전등이었다. 어디를 비출지는 내가 정했다.

### 6.1. Theme 6 다크 모드 "검은 직사각형"
**내가 알아챈 것.** 다크 모드 + info-active 상태에서, 화면 중앙에 단단한 직사각형 경계가 자꾸 눈에 걸렸다. Matrix rain이 어두운 배경에 깨끗한 가장자리에서 멈춰 있었다.

**내 가설.** 두 가지 중 하나라고 봤다 — (a) 캔버스 자체가 컨테이너와 다른 fill을 가지고 있거나, (b) 어떤 CSS transform이 가시적 바운딩 박스를 만들고 있거나. (a)는 캔버스가 `clearRect`를 쓰고 투명하다는 걸 알고 있었으므로 배제. 그러면 의심할 곳은 info-active 모드의 CSS 처리뿐이었다.

**내가 지시한 리서치.** AI에게 다음을 요청:
1. info-active 모드에서 캔버스의 computed style 출력.
2. `.ntm-wall.info-active`를 타깃하는 모든 CSS 규칙 나열.
3. CSS 효과를 하나씩 제거해보고, 어느 것을 제거했을 때 경계가 사라지는지 보고.

**내가 내린 결정.** AI가 `transform: scale(0.55)` + `clip-path` + `mask-image`의 상호작용임을 확인했다. 두 번의 CSS 레벨 수정(opacity만, 그 다음 radial-gradient mask)은 스크린샷에서 여전히 가시적 가장자리를 남겼다. 두 안 모두 거절하고, 구조적 결정을 내렸다 — **시각적 컨테인먼트에 CSS transform을 쓰지 말자. 타원형 마스크를 캔버스 그리기 코드 안으로 옮기자.** AI는 내가 명시한 것을 구현했다 — 매 프레임 입자별로 계산되는 타원형 알파 마스크, CSS transform 전혀 없음. 이게 동작한 이유는 캔버스가 이제 DOM에서 viewport 전체를 유지하고, 마스킹이 CSS 클리핑이 아니라 렌더링된 콘텐츠의 속성이 되었기 때문이다.

**내가 얻은 교훈.** CSS가 만들어낸 시각적 아티팩트를 더 많은 CSS로 부드럽게 만들 수 없을 때, 답은 보통 CSS에서 해결을 멈추는 것이다.

### 6.2. Theme 3 밀도 vs 성능
**내가 알아챈 것.** 6개 테마를 나란히 놓고 봤을 때, Theme 3가 다른 것들보다 눈에 띄게 약했다 — 희소하고, 무게감이 부족하고, 주제(수용소)와 어울리지 않았다.

**내 가설.** 문제는 입자 디자인이 아니라 입자 수였다. 메타포는 맞았다. 볼륨이 틀렸다.

**내가 지시한 리서치.** AI에게 (a) 현재 입자 수를 보고하고 (b) 늘리라고 했다. 첫 시도에서 880개(8 셸 × 110 입자)로 도달했다. 브라우저에서 테스트했고 즉시 프레임 드롭을 느꼈다.

**내가 내린 결정.** 렉을 받아들이는 대신 문제를 둘로 쪼갰다 — (i) GPU를 쏟아붓지 않으면서도 무게감 있게 읽히는 **목표 밀도**는 얼마인가, (ii) 외관을 잃지 않고 깎을 수 있는 **입자당 비용**은 무엇인가. AI에게 입자당 trail을 4개에서 2개로 줄이고(DOM 무게의 대부분을 만들고 있던 부분), 입자 수를 595(7 셸 × 85)로 정착시키라고 지시했다. DOM 노드 총합이 약 3,500에서 약 2,640으로 떨어졌다. 프레임 레이트가 회복되었다. 시각적 무게감은 여전히 원본의 약 1.5배.

**내가 얻은 교훈.** "입자 더 많이"와 "입자당 비용 더 높이"는 독립적으로 조절 가능한 다이얼이다. 밀도가 너무 비쌀 때, 단위 수를 줄이는 것보다 각 단위를 슬림하게 만드는 것이 거의 항상 더 싸다.

### 6.3. AI 같은 내러티브 정리
**내가 알아챈 것.** 우측 사이드바 내러티브들이 능숙하지만 일반적으로 들렸다. 특정한 운율이 있었다 — em-dash, 3개짜리 리스트, "X is not Y, it's Z" 반전, 그리고 모든 패널 끝의 빨간 이탤릭 격언. 소리내어 읽어봤더니 모두 같은 작가가 쓴 것 같았다.

**내가 지시한 리서치.** AI에게 "AI 같은 느낌 줄여줘"라고 하는 건 한 라운드의 미미한 개선만 만들어냈다. 그래서 직접 **AI 글의 실제 tells 리스트**를 찾아 나섰다. Hacker News의 커뮤니티 논의, PromptWorks 블로그의 AI 글 tells에 대한 글, r/webdev의 AI 텍스트 패턴 스레드들을 읽고, 구체적 체크리스트를 조립했다:
- "X is not Y, it's Z" 반전
- 3개 리스트 ("violence, obedience, falsehood")
- "In this work…" 메타 내러티브 도입부
- 결말 격언
- 균등한 문장 길이
- 추상 동사 ("delve", "explore", "navigate")

**내가 내린 결정.** 그 리스트를 그대로 AI에게 주고, 6개 내러티브 모두에 대해 각 항목별 패스를 돌리라고 지시했다. 결과는 사용 가능한 수준이었지만 — 더 중요하게, 그 리스트는 그 후의 모든 재작성에 대한 내 필터가 되었다.

**내가 얻은 교훈.** "느낌이 이상해"는 실행 가능하지 않다. 실제 작업은 무엇이 구체적으로 이상한지 식별하고, AI가 실행할 수 있는 형태로 그것을 적어두는 일이었다.

## 7. AI 협업 디자인에 대한 회고

프로젝트의 가장 명확한 교훈은 **AI는 디자이너가 아니라 제도공(draftsman)**이라는 점이다. 강한 스케치를 주면, 훌륭한 선을 그어준다. 약한 스케치를 주면, 아무것도 아닌 것에 대한 훌륭한 선을 그어준다. 규율은 "이걸 구현할 수 있나?"에서 "내가 정말 무엇을 원하는지 아는가?"로 이동한다.

기록할 만한 세 가지 관찰:

1. **신뢰하되 검증하라는 선택이 아니다.** AI는 존재하지 않는 함수(`onKey`)를 자신 있게 호출했고, 실제로 렌더링되지 않은 변경을 자신 있게 설명했고, 단독으로는 작동하지만 컨텍스트에서는 깨지는 코드를 자신 있게 만들었다. 브라우저 기반 검증 — 실제 스크린샷 찍기, 실제 computed style 읽기 — 가 코드 리뷰만으로는 놓치는 모든 것을 잡았다.

2. **구체적 지시가 추상적 지시를 능가한다.** "AI 같은 느낌 줄여줘"는 한 라운드의 미미한 개선만 만들어냈다. "em-dash 빼고, 결말 인용구 빼고, 'isn't X, it's Y' 패턴 죽여"가 실제 결과를 만들었다. 구체성이 곧 레버리지다.

3. **협업은 비대칭이다.** 인간은 키 입력의 약 5%, 결정의 약 95%를 기여했다. AI는 그 반대를 기여했다. 두 절반 모두 필요하다. 어느 한 절반도 충분하지 않다.

## 8. 리서치와 레퍼런스

이 프로젝트는 진공에서 발명되지 않았다. 다음은 빌드 중 실제로 참고된 주요 출처, 레퍼런스, 기술 자료다.

### 8.1. 원전 텍스트 (솔제니친)
6개 테마에 삽입된 인용구는 네 개의 텍스트에서 가져왔으며, 출판된 영어 번역본을 기준으로 한다:

- **"Live Not by Lies"** (Жить не по лжи, 1974) — 거짓에의 개인적 비참여를 다룬 핵심 선언문. *"let it not rule through me"* 구절이 Theme 5와 6의 중심.
- **The Gulag Archipelago** (Архипелаг ГУЛАГ, 1973) — 특히 1부 4장 ("The Bluecaps")의 "선이 모든 심장을 관통한다" 구절이 Theme 4 (Conscience)의 중심.
- **One Day in the Life of Ivan Denisovich** (Один день Ивана Денисовича, 1962) — 슈호프의 "3,653일" 구절이 Theme 3 (Gulag)의 중심.
- **Nobel Lecture** (1970) — 거짓에 맞서는 무기로서의 예술이라는 프레임. 작품 전반에 간접적으로 참조.

### 8.2. 어원
Theme 4 (Conscience)는 두 언어에서의 단어 어원에 직접 기반한다:
- 그리스어: **συν + εἶδος** (*syn + eidos* — "함께-앎")
- 라틴어: **con-scientia** ("함께-알기")
두 어원 모두 양심을 본질적으로 관계적인 것 — 자아가 자아를 보는, 결코 고독하지 않은 것 — 으로 프레임한다. 이것이 동심원 링이 함께 회전하는 시각적 선택을 만들었다.

### 8.3. 시각 / 미학 레퍼런스
- **Brutalist 웹 디자인** — 광범위하게는 brutalistwebsites.com 아카이브와 Brutalism Awards 계보. 모노스페이스, 노출 그리드, 강조색만 사용, 장식 없음의 미학적 어휘.
- **냉전기 redaction 문서** — 비밀 해제된 CIA/FOIA 아카이브 스캔(National Security Archive, gwu.edu/nsarchive). █ 블록 처리는 이들로부터 직접 모델링.
- **Matrix Rain (1999)** — Theme 6의 정전 시각 레퍼런스. 가타카나 글리프에서 괄호-눈으로 전치.
- **CRT 스캔라인 셰이더** — Theme 2의 `uIsLightMode` 셰이더 패턴은 Shadertoy CRT 효과 데모 계보.
- **JetBrains Mono** + **Inter** + **Noto Sans** — 타이포 스택. JetBrains Mono가 문서/redaction 톤을 담당, Inter가 UI를 담당.

### 8.4. 기술 문서
- **Three.js docs** (threejs.org/docs) — Theme 1, 4의 GLB 로더, 커스텀 셰이더, 포스트프로세싱 파이프라인(UnrealBloomPass)에 광범위하게 사용.
- **MDN Canvas API** (developer.mozilla.org) — Theme 5, 6의 레퍼런스. 특히 DPR-aware 렌더링, 오프스크린 캔버스를 통한 스프라이트 캐싱, `globalAlpha` 컴포지팅.
- **MDN CSS Grid + `:has()` 의사 클래스** — 8-column 메인 그리드와 호버-블러 형제 패턴.
- **CSS Tricks "Complete Guide to Grid"** — 사이드바의 grid-column 배치를 위해 참조.
- **WebGL Fundamentals** (webglfundamentals.org) — 셰이더 uniform 배관용.

### 8.5. AI 협업 작업에 대한 논의
직접 인용보다는 활발한 독해를 통해 추출된 일반 커뮤니티 컨센서스:
